## 1. 单局状态机

### 1.1 顶层状态转移图

```
┌─────────────┐     选择主角      ┌─────────────┐     确认配置      ┌──────────────────────────┐
│ HERO_SELECT │ ───────────────▶ │   TAVERN    │ ───────────────▶ │        RUNNING           │
│  (主角选择)  │                  │  (酒馆集结)  │                 │      (1~30回合推进)       │
└─────────────┘                  └─────────────┘                 └──────────────────────────┘
                                                                           │
                                                          第30回合节点执行完毕
                                                                           ▼
                                 ┌─────────────┐     终局战结束    ┌──────────────────────────┐
                                 │  SETTLEMENT │ ◀────────────── │      FINAL_BATTLE        │
                                 │  (终局结算)  │                 │      (终局战)             │
                                 └─────────────┘                 └──────────────────────────┘
                                       │
                                       ▼
                              结算完成 → 返回主菜单 [修正:M4: Phase 1不包含排行榜]
```


### 1.2 状态说明

| 状态 | 职责 | 允许的用户操作 |
|------|------|---------------|
| `HERO_SELECT` | 展示3名可选主角（勇者/影舞者/铁卫），等待玩家选择 | 选择主角、查看主角详情 |
| `TAVERN` | 展示默认解锁的6名伙伴，让玩家选择首发2人组成初始队伍（1主角+2伙伴） | 选择/取消伙伴、确认出发 |
| `RUNNING` | 30回合养成循环的核心状态，每回合经历"节点生成→选择→执行" | 三选一、执行节点交互 |
| `FINAL_BATTLE` | 第30回合节点执行完毕后自动进入终局战 | 观看战斗过程 |
| `SETTLEMENT` | 根据终局战结果生成斗士档案，计算评分 | 查看档案、确认结束 |

### 1.3 RUNNING阶段的子状态机

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              RUNNING 状态                                    │
│                                                                             │
│   ┌───────────────┐    玩家选择     ┌───────────────┐    节点逻辑结束     │
│   │ NODE_SELECT   │ ─────────────▶ │ NODE_EXECUTE  │ ─────────────────▶  │
│   │  (生成3选项)   │                │  (执行节点逻辑) │                     │
│   └───────────────┘                └───────────────┘                     │
│          ▲                                                                  │
│          │                        ┌───────────────┐                       │
│          └─────────────────────── │ TURN_ADVANCE  │                       │
│             生成下一回合选项       │  (回合推进)    │                       │
│                                   └───────────────┘                       │
│                                                                             │
│   状态转移规则：                                                             │
│   - 第1回合进入RUNNING时自动处于NODE_SELECT                                   │
│   - NODE_EXECUTE完成后检查当前回合数：                                         │
│     · 若 turn < 30：进入TURN_ADVANCE，turn++，回到NODE_SELECT                 │
│     · 若 turn == 30：进入FINAL_BATTLE                                        │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.4 回合推进规则

```
回合推进伪流程（每回合严格执行）：

1. 检查固定节点触发条件（见1.5固定节点触发表）
   └─ 若当前回合在固定节点表中，强制覆盖3个选项为对应节点
2. 若非常规固定节点回合：
   └─ 从节点池随机抽取3个选项（见第2章节点池算法）
3. 玩家三选一
4. 执行选中节点的逻辑（锻炼/战斗/商店/救援/PVP检定）
5. 节点执行完毕后：
   a. 更新GameState（金币/属性/伙伴状态等）
   b. 保存局内快照（见第7章存档设计）
   c. turn计数器+1
6. 循环至步骤1
```

## 2. 节点池生成算法

### 2.1 节点类型定义

| 节点类型 | 类型ID | 说明 |
|----------|--------|------|
| 锻炼 | `TRAIN` | 五属性之一的基础属性提升 |
| 普通战斗 | `BATTLE` | 与标准强度敌人战斗，获得金币[已对齐: 规格书5.1节] |
| 精英战 | `ELITE` | 与高强度敌人战斗，胜利奖励3选1（强力BUFF/大量金币/稀有道具），失败=本局结束 [已对齐: 规格书5.2节] |
| 商店 | `SHOP` | 购买主角升级或伙伴升级 |
| 救援 | `RESCUE` | 仅在第5/15/25回合触发，3选1新伙伴 |
| PVP检定 | `PVP` | 仅在第10/20回合触发，对战AI对手 |
| 终局战 | `FINAL` | 仅在第30回合触发 |

> **普通战斗节点规格** [已对齐: 规格书5.1节]：
> - 敌方根据当前回合和主角属性动态生成
> - 战斗模式：简化快进，2-3秒
> - 胜利奖励：金币（基于敌人难度和主角属性）
> - 失败惩罚：消耗生命值（基于属性差距），但**生命不会降至0**
> - 无失败风险，但损耗影响后续规划

> **精英战节点规格** [已对齐: 规格书5.2节]：
> - 出现概率：普通回合选项中随机出现（前期低、后期高）
> - 敌方：5种预设精英敌人模板（5种难度层级）
> - 战斗模式：标准播放，15-25秒
> - 胜利奖励：稀有奖励3选1（强力BUFF / 大量金币 / 稀有道具）
> - 失败惩罚：**本局结束**（唯一淘汰途径）
> - 可选放弃：玩家可选择跳过精英战

> **PVP检定节点规格** [已对齐: 规格书5.3节]：
> - 触发：第10/20回合固定触发
> - 对手：根据当前表现匹配的同水平AI对手
> - 战斗模式：标准播放
> - 失败惩罚：**不淘汰本局**，但影响奖励档位和后续难度
> - 累积惩罚：连续PVP失败叠加负面效果（生命上限降低、后续敌人变强）

### 2.3 三选一抽选算法

```
算法：每回合生成3个选项
输入：当前回合数 turn，连续无战斗计数器 no_battle_streak
输出：3个节点选项 [option_1, option_2, option_3]

步骤1：确定阶段
  若 turn ∈ [1,9]：phase = EARLY
  若 turn ∈ [10,19]：phase = MID
  若 turn ∈ [20,29]：phase = LATE

步骤2：检查保底机制
  若 no_battle_streak >= [待确认：规格书未明确保底阈值]：
    强制至少1个选项为战斗类型（BATTLE或ELITE）[待确认：规格书未明确保底细节]
    （注：锻炼和商店选项相应减少）

步骤3：权重抽选（带放回）[待确认：规格书未明确是否放回]
  从当前phase的权重配置表中，按相对权重进行3次独立抽选
  每次抽选一个节点类型，放回后抽下一次

  选项去重规则[待确认：规格书未明确去重规则]：
  - 方案A：允许3个选项完全相同
  - 方案B：同类型最多出现2次
  - 方案C：3个选项必须类型不同

步骤4：生成具体节点内容
  对每个抽中的节点类型，生成具体的节点实例：
  - TRAIN：随机从五属性（体魄/力量/敏捷/技巧/精神）中选1个[待确认：规格书未明确是否允许重复属性]
  - BATTLE：从普通战斗敌人池中随机选1个
  - ELITE：从精英战斗敌人池中随机选1个
  - SHOP：生成商店节点（具体商品见第4章）

步骤5：返回3个选项
```

## 3. 锻炼系统计算流程

### 3.1 五属性定义 [已对齐: 规格书3.3节]

| 属性 | 编码 | 说明 | 核心战斗作用 | 影响方向 |
|------|:----:|------|-------------|---------|
| 体魄 | 1 | 生命值/生存能力 | 生命/防御/抗伤 | 防御向 |
| 力量 | 2 | 物理攻击力 | 普攻伤害/物理技能/破防 | 进攻向 |
| 敏捷 | 3 | 速度/闪避 | 先手/闪避/连击/行动频率 | 机动向 |
| 技巧 | 4 | 暴击/命中 | 命中/暴击/技能触发率 | 技术向 |
| 精神 | 5 | 特殊防御/技能效果 | 抗性/低血爆发/终盘表现 | 辅助向 |

### 3.2 属性熟练度阶段定义 [已对齐: 规格书4.5节]

> **核心机制**：属性熟练度提供锻炼时的**额外加成**，锻炼次数越多，熟练度阶段越高，加成越高。
> 注意：这与"边际递减"是两个独立系统——熟练度加成**递增**，而边际递减在单项属性超过总投入60%时触发。

| 阶段 | 阶段ID | 锻炼次数区间 | 熟练度加成 | 说明 |
|------|:------:|:-----------:|:---------:|:------|
| 生疏 | `NOVICE` | 0 | +0 | 初始状态，无加成 |
| 熟悉 | `FAMILIAR` | 1-3 | +2 | 初有心得，锻炼时额外+2 |
| 精通 | `PROFICIENT` | 4-6 | +4 | 熟练掌握，锻炼时额外+4 |
| 专精 | `MASTER` | ≥7 | +5 | **保守设计**（原方案为+9，初版降至+5，测试后再决定是否加强）[已对齐: 规格书1.3节决策#5] |

> **加成应用方式**：
> ```
> 单次锻炼最终增长 = 基础增长值 + 熟练度加成(阶段)
> ```
> 其中基础增长值由配置表决定 [待确认：规格书未明确基础增长值具体数值]。
> 例如：基础增长值为+5，处于精通阶段(+4)，则单次锻炼总增长 = 5 + 4 = +9。

### 3.3 边际递减机制 [已对齐: 规格书4.5节]

```
边际递减触发条件：
  当单项属性的锻炼次数占总锻炼次数的比例超过 60% 时，
  该属性后续锻炼的基础增长值递减 20%。
  
  即：if attr_train_count[X] / total_train_count > 0.6:
        base_gain_effective = base_gain * 0.8

注意：
  - 递减仅影响基础增长值，不影响熟练度加成
  - 熟练度加成部分不受边际递减影响
  - 副属性锻炼的50%熟练度共享计数也不影响此项计算
```

### 3.4 副属性50%熟练度共享机制 [已对齐: 规格书4.5节]

```
共享规则：
当玩家锻炼属性X时，与X关联的副属性Y获得50%的熟练度计数增长。
注意：仅增加熟练度计数，不直接增加属性值。

共享映射表 [已对齐: 规格书4.5节]：
  锻炼体魄 → 精神熟练度计数 +0.5
  锻炼力量 → 技巧熟练度计数 +0.5
  锻炼敏捷 → 力量熟练度计数 +0.5
  锻炼技巧 → 敏捷熟练度计数 +0.5
  锻炼精神 → 体魄熟练度计数 +0.5

共享熟练度的效果：
  当副属性Y因共享累计到整数次数时，其train_count[Y] +1。
  这意味着副属性Y可以更快地达到更高熟练度阶段，
  从而在直接锻炼该属性时享受更高的熟练度加成。

示例 [已对齐: 规格书4.5节]：
  玩家连续4次锻炼体魄：
  - 体魄：train_count +4，属性值 +4次基础增长 +4次熟练度加成
  - 精神：共享 +2.0次熟练度计数，train_count[精神] +2
  当之后玩家选择锻炼精神时，精神的初始熟练度阶段为熟悉（1-3次），享受+2加成。
```

### 3.5 锻炼执行流程 [已对齐: 规格书4.5节]

```
流程：玩家选择锻炼节点（指定属性X）

步骤1：读取GameState中属性X的累计锻炼次数 train_count[X]

步骤2：判定当前熟练度阶段
  若 train_count[X] == 0：stage = NOVICE, bonus = +0
  若 train_count[X] ∈ [1, 3]：stage = FAMILIAR, bonus = +2
  若 train_count[X] ∈ [4, 6]：stage = PROFICIENT, bonus = +4
  若 train_count[X] ≥ 7：stage = MASTER, bonus = +5
  [已对齐: 规格书4.5节]

步骤3：获取基础增长值 base_gain
  base_gain 来自配置表 [待确认：规格书未明确基础增长值具体数值]

步骤4：检查边际递减
  total_train = 所有属性锻炼次数之和
  若 train_count[X] / total_train > 0.6：
    base_gain = base_gain * 0.8  （递减20%）
  [已对齐: 规格书4.5节]

步骤5：计算最终增长值
  final_gain = base_gain + bonus（熟练度加成不受边际递减影响）

步骤6：更新属性值
  attr_X_new = attr_X_old + final_gain

步骤7：更新锻炼计数
  train_count[X]++

步骤8：更新副属性共享熟练度
  根据3.4节的共享映射表，为副属性Y增加0.5次共享计数
  若累计到整数，train_count[Y] += 对应整数值
  [已对齐: 规格书4.5节]

步骤9：记录本次锻炼日志
  { turn, attr, stage, base_gain, bonus, final_gain, before, after }
```

### 3.6 手工验证案例：连续10次锻炼力量 [已对齐: 规格书4.5节]

### 3.7 边际递减验证案例 [已对齐: 规格书4.5节]

  3. 战斗结果处理：
     - 胜利：无额外惩罚，正常继续
     - 失败：不淘汰本局，但施加对应惩罚 [已对齐: 规格书5.3节]
  4. 记录PVP日志
     { turn, opponent_level, result, damage_dealt, damage_taken }

PVP失败惩罚分层 [已对齐: 规格书5.3节]：
┌───────────┬──────────────────────────────────────────────┐
│ PVP次数   │ 失败惩罚                                      │
├───────────┼──────────────────────────────────────────────┤
│ 第10回PVP │ 降低当前奖励档位                              │
│ 第20回PVP │ 大幅降低奖励 + 后续敌人属性 +15%              │

└───────────┴──────────────────────────────────────────────┘

### 6.3 评分公式 [已对齐: 规格书6.3节]（已定稿）

> **总分公式** [已对齐: 规格书6.3节]：
> ```
> 总分 = 终局战表现分 × 0.4
>      + 养成效率分 × 0.2
>      + PVP表现分 × 0.2
>      + 流派纯度分 × 0.1
>      + 连锁展示分 × 0.1
> ```

#### 1. 终局战表现分（权重40%，满分100） [已对齐: 规格书6.3节]

| 子项 | 计算方式 | 满分 |
|:---|:---|:---:|
| 终局战胜利 | 胜利 = 50分，失败 = 0分 | 50 |
| 剩余生命比例 | （终局战后剩余生命 / 最大生命）× 30 | 30 |
| 终局战造成伤害 | （实际造成伤害 / 敌方最大生命）× 20 | 20 |
| **小计** | | **100** |

```
终局战表现分 = (终局战胜 ? 50 : 0)
             + (remaining_hp / max_hp) * 30
             + (damage_dealt / enemy_max_hp) * 20
[已对齐: 规格书6.3节]
```

#### 2. 养成效率分（权重20%，满分100） [已对齐: 规格书6.3节]

| 子项 | 计算方式 | 满分 |
|:---|:---|:---:|
| 总属性成长 | （终局五维总和 / 初始五维总和 - 1）× 30 | 30 |
| 金币使用效率 | （实际用于升级的金币 / 获得的总金币）× 30 | 30 |
| 属性均衡度 | 五维标准差的倒数 × 20（越均衡分越高） | 20 |
| 精英战胜率 | （精英战胜利次数 / 精英战总次数）× 20 | 20 |
| **小计** | | **100** |

```
养成效率分 = (sum(final_attrs) / sum(initial_attrs) - 1) * 30
           + (gold_spent / gold_earned) * 30
           + (1 / std_dev(final_attrs)) * 20
           + (elite_wins / elite_total) * 20
[已对齐: 规格书6.3节]
```

#### 3. PVP表现分（权重20%，满分100） [已对齐: 规格书6.3节]

| 子项 | 计算方式 | 满分 |
|:---|:---|:---:|
| 第10回PVP | 胜利 = 40分，失败 = 15分，跳过 = 0分 | 40 |
| 第20回PVP | 胜利 = 40分，失败 = 15分，跳过 = 0分 | 40 |
| PVP造成伤害 | 两场PVP总伤害 / 参考值 × 20 | 20 |
| **小计** | | **100** |

```
PVP表现分 = pvp_10_score + pvp_20_score + pvp_damage_score

其中：
  pvp_10_score = 胜 ? 40 : (负 ? 15 : 0)   // 跳过=0
  pvp_20_score = 胜 ? 40 : (负 ? 15 : 0)   // 跳过=0
  pvp_damage_score = (pvp_10_damage + pvp_20_damage) / reference_value * 20
[已对齐: 规格书6.3节]
```

> **PVP失败惩罚分层** [已对齐: 规格书5.3节]：
> - 第10回PVP失败：降低当前奖励档位
> - 第20回PVP失败：大幅降低奖励 + 后续敌人属性+15%
> - 终局PVP：影响通关评价和斗士档案评分

#### 4. 流派纯度分（权重10%，满分100） [已对齐: 规格书6.3节]

| 子项 | 计算方式 | 满分 |
|:---|:---|:---:|
| 主属性占比 | 最高属性 / 五维总和 × 50 | 50 |
| 技能触发次数 | 常规技能+必杀技总触发次数 / 参考值 × 30 | 30 |
| 伙伴协同 | 伙伴援助总触发次数 / 参考值 × 20 | 20 |
| **小计** | | **100** |

```
流派纯度分 = (max_attr / sum(all_attrs)) * 50
           + (skill_trigger_count / reference_value) * 30
           + (partner_assist_count / reference_value) * 20
[已对齐: 规格书6.3节]
```

#### 5. 连锁展示分（权重10%，满分100） [已对齐: 规格书6.3节]

| 子项 | 计算方式 | 满分 |
|:---|:---|:---:|
| 最大连锁段数 | 单场最高CHAIN数 × 10（上限4段 = 40分） | 40 |
| 总连锁次数 | 全战斗连锁总次数 / 参考值 × 30 | 30 |
| 伙伴援助总次数 | 全战斗伙伴援助触发总次数 / 参考值 × 30 | 30 |
| **小计** | | **100** |

```
连锁展示分 = min(max_chain, 4) * 10
           + (total_chain_count / reference_value) * 30
           + (total_partner_assist / reference_value) * 30
[已对齐: 规格书6.3节]
```

> **注意**：连锁触发最大链长为4段（规格书4.4节），因此max_chain上限为4。

#### 总分计算公式汇总 [已对齐: 规格书6.3节]

```
final_score = 终局战表现分 * 0.4
            + 养成效率分 * 0.2
            + PVP表现分 * 0.2
            + 流派纯度分 * 0.1
            + 连锁展示分 * 0.1
```

#### 评级标准 [已对齐: 规格书6.3节]（已定稿）

| 评级 | 总分范围 | 说明 |
|:---:|:---:|:---|
| **S** | ≥ 90 | 完美通关，构筑和操作都达到极高水平 |
| **A** | 75-89 | 优秀通关，构筑合理，有少量优化空间 |
| **B** | 60-74 | 良好通关，完成基本目标 |
| **C** | 40-59 | 勉强通关，构筑有明显缺陷 |
| **D** | < 40 | 失败或未通关 |

> **分数设计意图** [已对齐: 规格书6.3节]：
> - **终局战40%权重**：终局战是整局养成的最终验证，最重要的分数来源
> - **PVP20%权重**：PVP是中期检定，表现好坏直接影响总评
> - **养成效率20%权重**：鼓励合理规划资源，而非无脑堆单一属性
> - **流派纯度10%**：鼓励专精一个build方向，而非平均分配
> - **连锁展示10%**：鼓励触发华丽的伙伴连锁，提升观赏性

### 6.4 固定化标记机制 [已对齐: 规格书4.6节]

```
is_fixed 字段语义 [已对齐: 规格书4.6节]：
  - true：该档案已完成终局结算，数据不可再修改
  - false：该档案对应进行中的对局（用于局内存档）

固定化时机 [已对齐: 规格书4.6节]：
  - 终局结算流程完成时，自动将 is_fixed 设为 true
  - 此后该档案仅可读取，不可写入
  - 锁定所有属性值
  - 保留版本号用于向后兼容

用途 [已对齐: 规格书4.6节]：
  - 防止历史档案被意外篡改
  - 排行榜仅展示 is_fixed = true 的档案
  - 进行中的存档 is_fixed = false，读档后可继续游戏
```
