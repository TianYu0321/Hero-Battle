### Bug 1：继续游戏楼层回到第 1 层

**根因**
`run_controller.gd` → `continue_from_save()` 里用 `snapshot.current_floor`，但 `_save_at_floor_entrance()` 调的是 `_run.to_dict()`，RuntimeRun 里字段名叫 `current_turn`。字段名不一致 → `snapshot.current_floor` 默认值 0 → `_run.current_turn = 0` → 游戏从第 1 层重新开始。

**更深层问题**
`_save_at_floor_entrance()` 直接序列化运行时对象（`_run.to_dict()` + `_hero.to_dict()`），没有独立存档数据层，字段名、版本、结构完全取决于运行时对象的实现。每次修改 RuntimeRun/RuntimeHero 都可能破坏存档兼容性。

**修复位置**
- `scripts/systems/run_controller.gd`

**修复步骤（二选一）**

方案 A（快速修）：把 `continue_from_save()` 里的 `snapshot.current_floor` 统一改为 `snapshot.current_turn`：
```gdscript
_run.current_turn = snapshot.current_turn  # 原来写的是 snapshot.current_floor
```

方案 B（推荐，长期）：`_save_at_floor_entrance()` 不再直接调 `_run.to_dict()`，而是显式构造独立存档字典，统一字段名：
```gdscript
var save_data := {
    "current_floor": _run.current_turn,
    "hero_config_id": _run.hero_config_id,
    "gold": _run.gold_owned,
    "hero": {
        "current_hp": _hero.current_hp,
        "max_hp": _hero.max_hp,
        "current_vit": _hero.current_vit,
        "current_str": _hero.current_str,
        "current_agi": _hero.current_agi,
        "current_tec": _hero.current_tec,
        "current_mnd": _hero.current_mnd,
        "training_counts": _hero.training_counts.duplicate(),
    },
    "partners": _get_partner_dicts(),
    "node_history": _run.node_history.duplicate(true),
    "battle_win_count": _run.battle_win_count,
    "elite_win_count": _run.elite_win_count,
    "event_forecast_charges": forecast_system.get_charges() if forecast_system != null else 0,
}
SaveManager.save_run_state(save_data, true)
```
`continue_from_save()` 同步改为读这些统一字段名，不再依赖 `RunSnapshot` 的 from_dict 映射。

### Bug 2：PVP 与爬塔战斗系统不一致

**根因 A：爬塔角色不动（无动画）**
`run_main.gd` → `_on_battle_ended()` 走爬塔分支时，`battle_result` 里没有 `hero_sprite_path`/`enemy_sprite_path`。`battle_animation_panel.gd` → `start_battle()` 里调用 `_load_sprite(hero_art, "")`，空路径直接 return，`AnimatedSprite2D.sprite_frames` 保持 null。后续 `_play_anim()` 检查 `sprite_frames == null` 直接 return，角色永远不会播放 attack/hurt/dead 动画。

**修复位置**
- `scenes/run_main/run_main.gd` → `_on_battle_ended()` 爬塔分支

**修复步骤**
在 `battle_result` 字典被传入 `battle_animation_panel.start_battle()` 之前，补上 sprite_path：
```gdscript
# 在构造 battle_result 后、调用 start_battle 前插入：
var hero_sprite_path: String = ""
match hero_config_id:
    1: hero_sprite_path = "res://assets/characters/warrior/hero_frames.tres"
    2: hero_sprite_path = "res://assets/characters/shinobi/hero_frames.tres"
    # 其他英雄继续补
battle_result["hero_sprite_path"] = hero_sprite_path
battle_result["enemy_sprite_path"] = "res://assets/characters/enemies/enemy_frames.tres"  # 按敌人配置动态取
```

**根因 B：PVP 只打一回合后续空白**
`run_main.gd` → `_on_battle_ended()` 调用 `start_playback()` 时传入 `battle_result.get("turns_elapsed", 0)` 作为 `total_rounds`。如果 BattleEngine 返回的 `turns_elapsed` 为 0/1（或 recorder 实际有 5 回合但 turns_elapsed 写错），`_sim_total_rounds` 被设为这个低值，`_current_round` 增加到 2 时触发 `should_end` → 直接结算。另外 `_play_next_turn()` 里对 recorder 事件的 `has()` 查找用 int key，但 recorder 返回的 key 可能是 String，导致所有事件被跳过。

**修复位置**
- `scenes/run_main/battle_animation_panel.gd` → `start_playback()`

**修复步骤**
1. `_sim_total_rounds` 用 recorder 真实回合数，不用传入的 `total_rounds`：
```gdscript
# start_playback() 末尾，在 _play_next_turn() 之前：
_sim_total_rounds = maxi(_turn_keys.max() if _turn_keys.size() > 0 else 0, 1)
# 或保底：
_sim_total_rounds = maxi(_real_max_turn, 1)
```
2. 统一 key 类型为 int：
```gdscript
var raw_events: Dictionary = _recorder.get_events_by_turn()
for k in raw_events.keys():
    var ik: int = int(k)
    _events_by_turn[ik] = raw_events[k]
    _turn_keys.append(ik)
    _real_max_turn = maxi(_real_max_turn, ik)
_turn_keys.sort()
```

**长期建议**
两套战斗入口（`start_playback` vs `start_battle`）应统一为一套：爬塔战斗也用 BattleEngine 生成 recorder，然后走 `start_playback`。删掉 `start_battle` 的模拟模式，避免两套 HP 更新逻辑、两套事件处理、两套动画触发。所有战斗动画统一从 recorder 的事件流播放。
