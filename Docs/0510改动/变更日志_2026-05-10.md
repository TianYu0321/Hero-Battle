# 变更日志 - 2026-05-10

## 概述

本次变更主要修复了 `test_full_run.tscn` 全流程测试中暴露的**8项验收失败**问题，并清理了多个编译错误和运行时逻辑缺陷。

**验收结果**：
- ultimate_triggered: 通过
- chain_triggered: 通过
- assist_triggered: 通过
- rescue_triggered: 通过
- elite_triggered: 通过
- run_ended信号: 收到
- archive生成: 非空
- 评分等级: B

## 问题根因

`test_full_run.gd` 测试脚本在处理需要 UI 交互的节点（训练、救援、商店）时存在根本性缺陷：

1. `select_node()` 对 `requires_ui_selection == true` 的节点会 `early-return`，状态停留在 `RUNNING_NODE_EXECUTE`
2. 测试脚本未调用对应的 UI 回调（`select_training_attr()` / `select_rescue_partner()`）
3. 随后调用 `advance_turn()` 因 `_state != TURN_ADVANCE` 而直接 return
4. **回合永远卡住不推进**，导致30回合循环实际只执行了第1层，终局战/结算/信号全部无法触发

## 详细改动

### 1. 测试脚本重写（核心修复）

**文件**: `scenes/test/test_full_run.gd`

**问题与修复**：
- `select_node()` 后无 UI 回调处理：增加状态检查，若卡在 `RUNNING_NODE_EXECUTE`，根据 `node_type` 调用对应回调：`TRAINING`->`select_training_attr()`、`RESCUE`->`select_rescue_partner()`、`SHOP/OUTING`->强制 `_finish_node_execution()` 跳过
- 救援 PID 获取错误：`candidates[0].get("partner_config_id", 0)` 改为 `candidates[0].get("partner_id", "0")`（候选字典中无 `partner_config_id` 字段）
- 回合卡住无兜底：循环开始时增加状态恢复，若 `run_state == RUNNING_NODE_EXECUTE`，强制调用 `_finish_node_execution({"success": true})` + `advance_turn()`
- 永远选不到 OUTING：`_choose_option` 逻辑调整，属性总和 >=100 后，若 elite 未触发，优先选择 OUTING 节点尝试触发精英战
- node_type 映射错误：修正注释，`3=REST`(原误标为 ELITE)、`4=OUTING`(原误标为 SHOP)

**关键代码片段**（状态恢复与 UI 处理）：
```gdscript
# 循环开始时：若上一次回合卡住，强制推进
if run_state == RUNNING_NODE_EXECUTE:
    _rc.call("_finish_node_execution", {"success": true})
    _rc.advance_turn()

# select_node() 后：根据 node_type 调用 UI 回调
match node_type:
    NodePoolSystem.NodeType.TRAINING:
        _rc.select_training_attr(attr)
    NodePoolSystem.NodeType.RESCUE:
        var pid = candidates[0].get("partner_id", "0")
        _rc.select_rescue_partner(int(pid))
    NodePoolSystem.NodeType.SHOP, NodePoolSystem.NodeType.OUTING:
        _rc.call("_finish_node_execution", {"success": true})
```

### 2. RunController - PVP 执行与救援推进

**文件**: `scripts/systems/run_controller.gd`

**问题与修复**：
- PVP 节点未实际执行：`_process_node_result()` 新增 `elif _pending_node_type == NodePoolSystem.NodeType.PVP_CHECK:` 分支，调用 `PvpDirector.execute_pvp()` 并记录结果
- `select_rescue_partner()` 未推进回合：在 `rescue_partner()` 调用后补充 `_finish_node_execution(_pending_result)`，使救援节点能正常进入 `TURN_ADVANCE`
- PVP 回合 `node_type` 硬编码错误：`_generate_node_options()` 中 `"node_type": 6` 改为 `NodePoolSystem.NodeType.PVP_CHECK`（6 是 SHOP，7 才是 PVP_CHECK）
- `_get_front_enemy()` 返回 null：新增 `_hero_to_battle_dict()` 辅助方法，为 `PvpDirector.execute_pvp()` 提供 `player_hero` 数据

### 3. NodeResolver - 信号转发与精英概率

**文件**: `scripts/systems/node_resolver.gd`

**问题与修复**：
- `node_resolved` 信号未到达测试脚本：`resolve()` 末尾增加 `EventBus.node_resolved.emit(_get_node_type_name(node_type), result)`，并新增 `_get_node_type_name()` 映射方法
- 精英战概率过低（5%），测试难以触发：`_OUTING_EVENTS` 中 `elite` 权重从 **5** 提高到 **25**，其他事件权重同步调整保持总和100

调整后的权重：
```gdscript
{"event": "rest",     "weight": 20}
{"event": "shop",     "weight": 15}
{"event": "trap",     "weight": 10}
{"event": "heal",     "weight": 10}
{"event": "special",  "weight": 20}
{"event": "elite",    "weight": 25}  // 原为 5
```

### 4. ConfigManager - 伙伴配置字段名错误

**文件**: `autoload/config_manager.gd`

**问题与修复**：
- `get_all_partner_config_ids()` 读取 `hero_id` 字段，但 `partner_configs.json` 中使用的是 `id` 字段：`cfg.get("hero_id", 0)` 改为 `cfg.get("id", 0)`
